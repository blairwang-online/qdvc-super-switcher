## 2. LLM response

Here's a script that maps the index to the Nth window in taskbar order and activates it via xdotool.

```python
#!/usr/bin/env python3
import subprocess
import sys

def sh(cmd):
    return subprocess.run(cmd, capture_output=True, text=True).stdout

def get_current_desktop():
    # _NET_CURRENT_DESKTOP
    out = sh(["xprop", "-root", "_NET_CURRENT_DESKTOP"])
    return int(out.split("=")[1].strip())

def get_window_list():
    # _NET_CLIENT_LIST_STACKING is stacking order; we want creation/taskbar order
    # _NET_CLIENT_LIST gives initial-mapping order, which matches taskbar order on MATE
    out = sh(["xprop", "-root", "_NET_CLIENT_LIST"])
    # e.g. "_NET_CLIENT_LIST(WINDOW): window id # 0x..., 0x..., ..."
    ids = out.split("#", 1)[1]
    return [w.strip() for w in ids.split(",") if w.strip()]

def get_long(wid, prop):
    out = sh(["xprop", "-id", wid, prop])
    if "=" not in out:
        return None
    return out.split("=")[1].strip()

def is_normal_window(wid):
    # Skip docks, panels, desktop, etc.
    wtype = get_long(wid, "_NET_WM_WINDOW_TYPE") or ""
    if "_NET_WM_WINDOW_TYPE_NORMAL" in wtype:
        return True
    # Many normal windows have no _NET_WM_WINDOW_TYPE set; accept those too
    if wtype == "" or wtype == "not found.":
        return True
    return False

def is_skip_taskbar(wid):
    state = get_long(wid, "_NET_WM_STATE") or ""
    return "_NET_WM_STATE_SKIP_TASKBAR" in state

def get_desktop(wid):
    val = get_long(wid, "_NET_WM_DESKTOP")
    if val is None or "not found" in val:
        return None
    try:
        return int(val.split()[0].split(",")[0])
    except ValueError:
        return None

def main():
    if len(sys.argv) != 2:
        sys.exit("usage: super_switcher.py N")
    try:
        n = int(sys.argv[1])
    except ValueError:
        sys.exit("argument must be an integer 1-9")
    if n < 1:
        sys.exit("argument must be >= 1")

    current = get_current_desktop()
    windows = []
    for wid in get_window_list():
        if not is_normal_window(wid):
            continue
        if is_skip_taskbar(wid):
            continue
        d = get_desktop(wid)
        # 0xFFFFFFFF (-1 / sticky) shows on all desktops
        if d is not None and d != current and d != 0xFFFFFFFF:
            continue
        windows.append(wid)

    if n > len(windows):
        return  # nothing in that slot
    target = windows[n - 1]
    subprocess.run(["xdotool", "windowactivate", target])

if __name__ == "__main__":
    main()
```

Setup: make it executable (`chmod +x super_switcher.py`), then in System → Preferences → Hardware → Keyboard Shortcuts add custom shortcuts binding `Super+1` to `/path/super_switcher.py 1`, and so on.

Notes: `_NET_CLIENT_LIST` gives initial-mapping (taskbar) order, which is what the Window List applet uses, so the numbering should match the panel. It filters to the current workspace plus sticky windows and skips taskbar-excluded windows. If your panel is set to show windows from all workspaces, drop the desktop-filtering block. If you'd rather avoid spawning many `xprop` calls per invocation, the same logic ports cleanly to `python-xlib`.

## 5. User request

If I drag-and-drop taskbar buttons around on MATE, their sequence in the taskbar changes, but they retain their old integer numbers for Super+n.

## 6. LLM response

The problem is that `_NET_CLIENT_LIST` reflects initial-mapping order and never updates when you drag taskbar buttons around — that reordering lives inside the Window List applet itself and isn't published to any root window property. So `xprop`/`xdotool` can't see it.

To track the visual order you have to read it from the applet's widget tree directly. On MATE the Window List is a GTK applet, and its buttons are laid out in the order shown. The practical way to get at that is through the accessibility (AT-SPI) layer, which exposes each task button as an accessible object in left-to-right order.The short answer: not from `_NET_CLIENT_LIST`, and not cleanly from any X property — the drag reorder lives entirely inside the Window List applet's GTK widget layout and is never published anywhere `xprop`/`xdotool` can read.

There are two practical routes.

**Route 1 — AT-SPI (reads the applet's actual visual order).** The task buttons are accessible objects laid out left-to-right in the panel, so walking the accessibility tree gives you the rearranged sequence. This requires accessibility to be enabled (`gsettings set org.mate.interface accessibility true`, then re-login) and the `gir1.2-atspi-2.0` package. The catch is that matching an accessible button back to an X window ID for `xdotool` is fiddly — task buttons expose a label (the window title), not the XID, so you match by title, which breaks with duplicate titles or grouping. It's the only way to get the true visual order, but it's brittle.

**Route 2 — track the reordering yourself.** Since the applet won't tell you its order, maintain your own. The first time the script sees a window it appends it to a saved list; you re-sort that list however you like. This doesn't auto-follow drags either, though.

Honestly, given the matching problems, the most robust approach that *does* follow your drags is to lean on the applet's own ordering through AT-SPI but activate by matching title to the `_NET_CLIENT_LIST` entries. Here's that version:A few honest caveats so you know what you're getting into:

The title-matching is the weak point. If two windows share a title (two terminals, two "Untitled Document" editors), or if you have grouping turned on so one button represents several windows, the match is ambiguous and you may activate the wrong window. Turning off "Group windows" in the applet's preferences avoids the grouping half of that problem.

Accessibility has to be enabled and the session restarted, which has minor side effects (some apps draw slightly differently, and you may see the occasional `at-spi-registryd` shutdown prompt). The AT-SPI role of the buttons can also vary by version — if `find_tasklist_buttons()` comes back empty, add `Atspi.Role.LABEL` or print the roles you find to see what your panel reports.

If all of that feels too fragile, the genuinely robust fix is to skip the panel applet entirely and define your *own* fixed ordering (e.g. the original `_NET_CLIENT_LIST` version, which is stable and unambiguous), accepting that it won't follow drags. The applet simply doesn't expose its order through any reliable channel, so there's an inherent trade-off between "follows drags" and "always picks the right window."

## 7. User request

OK, before we make any changes to the code, I would like to understand the numbering based on `_NET_CLIENT_LIST` a bit better.

Let's say I launch programs A, B, C, and D in that sequence (so they are now 1, 2, 3, 4 in _NET_CLIENT_LIST). I noticed that if I close program C, then program D now takes on position 3 in _NET_CLIENT_LIST. So I guess _NET_CLIENT_LIST is refreshing somehow.

Is there some action that the user can take in MATE itself to force a _NET_CLIENT_LIST refresh?

## 8. LLM response

You're conflating two different things, so let me separate them.

`_NET_CLIENT_LIST` isn't a list that "refreshes" on a user action. It's a property the window manager (Marco) rewrites every time the set of managed windows changes — i.e. whenever a window is created or destroyed. When you closed C, Marco rebuilt the property from its current client set, and since C was gone the list became A, B, D, and D's position shifted from 4 to 3. There's no caching and no staleness; it's always current as of the last map/unmap event. So there's nothing to "force" — it updates on its own the instant windows open or close, and that's the only thing that changes it.

What it does *not* do is reorder when you drag taskbar buttons, because dragging doesn't create or destroy a window — Marco's client set is unchanged, so the property is unchanged. That's the whole reason it can't follow your drag-and-drop.

So to your actual question: no, there is no user action in MATE that reorders `_NET_CLIENT_LIST` to match a rearranged taskbar. The only "action" that changes it is opening or closing windows, and that change reflects window lifetime, not panel position.

One subtlety worth knowing: the ordering within `_NET_CLIENT_LIST` is defined by the spec as *initial mapping order* — the order windows were first managed. Marco preserves that order and just removes entries as windows close (so survivors keep their relative order and shift down to fill gaps, exactly as you saw with D). It never promotes a window to a new slot based on anything you do at the panel. There's a sibling property, `_NET_CLIENT_LIST_STACKING`, which *does* reorder — but it tracks z-order (which window is on top), changing every time you focus or raise a window, which is even less like taskbar position.

If your real goal is "numbers that track the visual taskbar after I drag," neither property will ever give you that, for the structural reason above — the order you're rearranging exists only inside the applet's widget layout. The AT-SPI route remains the only thing that sees it.
